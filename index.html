Seven layers, two (or three) cross-cutting rails, and any number of workflows that weave through them. Flip between a personal setup and an enterprise one to see how the same stack changes shape.

A workflow isn't a single tool — it's a thread that picks one node per layer and zigzags between local/open and cloud/closed as it descends.

Product lens. In enterprise mode the hard part isn't the model — it's the Enterprise Knowledge layer: connecting siloed sources, doing permission-aware retrieval (answers respect each user's entitlements), and proving it with audit + DLP. The Governance frame wraps every layer because compliance isn't a feature you bolt on at the end.

So… what do I use when?

Answer a few questions and your thread falls out.

Is the data sensitive, regulated, or must it stay in-tenant?
Yes → enterprise thread. Private gateways (Bedrock/Azure/Vertex) keep frontier models in your VPC; retrieval is permission-aware; Governance is on. No, just personal-private → a local thread (Ollama + open model) is enough.
Is the task hard — real coding, deep reasoning, long agentic work?
Yes → frontier. Claude Code + a frontier model. Bulk & repetitive → a cheap open model, local or hosted (DeepSeek via OpenRouter).
Does it run on a schedule / unattended?
Yes → the Orchestration rail drives the thread automatically. Your stock scanner is exactly this — the amber zigzag.
Should the AI know my org's knowledge?
Yes → the Enterprise Knowledge layer: connectors into SharePoint/Confluence/Slack/warehouses, then permission-aware RAG over them. Never let retrieval bypass the asking user's access rights.
